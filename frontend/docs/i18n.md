# i18n — Internationalization

## Overview

The frontend uses **vue-i18n** (Composition API mode) with 3 supported languages: English, French, and Turkish. Language preference is persisted locally and synced with the backend.

## Supported Languages

| Code | Language | Locale File |
|------|----------|-------------|
| `en` | English | `src/i18n/locales/en.json` |
| `fr` | Francais | `src/i18n/locales/fr.json` |
| `tr` | Turkce | `src/i18n/locales/tr.json` |

## Architecture

```
src/i18n/
  index.ts          -> createI18n setup, resolveLocale()
  locales/
    en.json         -> English translations
    fr.json         -> French translations
    tr.json         -> Turkish translations
```

## Plugin Setup

**File:** `src/i18n/index.ts`

```ts
export const i18n = createI18n({
  legacy: false,         // Composition API mode
  globalInjection: true, // $t() available in all templates
  locale: resolveLocale(),
  fallbackLocale: 'en',
  messages: { en, fr, tr },
})
```

Registered in `main.ts` as `app.use(i18n)` — after Pinia, before Vue Router.

## Locale Resolution (Cold Start)

`resolveLocale()` uses a 3-layer fallback:

1. `localStorage.getItem('locale')` — user's saved preference
2. `navigator.language` — browser language (first 2 chars)
3. `'en'` — default fallback

Only `en`, `fr`, `tr` are accepted at each layer.

## Persistence

### Layer 1: localStorage

When the user changes language in Settings, it immediately:
- Updates `i18n.global.locale.value` (live preview)
- Writes to `localStorage.setItem('locale', lang)`

### Layer 2: Backend Sync

On "Save Settings", the frontend calls:
```
PATCH /users/:id/settings  { language: 'fr' }
```

On login/refresh, `checkAuth()` in the auth store reads the language from `GET /users/me` and re-applies it:
```ts
if (userData.settings?.language) {
  i18n.global.locale.value = userData.settings.language
  localStorage.setItem('locale', userData.settings.language)
}
```

This ensures language preference survives across devices.

## Settings UI

**File:** `src/components/user/SettingsSection.vue`

A `<select>` dropdown renders all supported languages with human-readable labels (English, Francais, Turkce). Changing the dropdown applies the language instantly (live preview before save).

## Usage in Components

All components use `<script setup lang="ts">` with `$t()` in templates:

```vue
<template>
  <h1>{{ $t('nav.home') }}</h1>
</template>
```

For script-side access:
```ts
import { useI18n } from 'vue-i18n'
const { t } = useI18n()
const label = t('common.save')
```

## Backend Contract

The backend needs to:

1. **Store** `language` in the user settings entity (string: `'en' | 'fr' | 'tr'`)
2. **Accept** `PATCH /users/:id/settings` with `{ language: string }`
3. **Return** `settings.language` in `GET /users/me` response

## Known Issue

`SUPPORTED_LANGUAGES` in `src/types/user.ts` lists 5 languages (`en`, `fr`, `tr`, `nl`, `ko`), but only 3 have locale files. The extra entries (`nl`, `ko`) appear in the settings dropdown but fall back to English at runtime. Either remove them from the type or add locale files if support is planned.
